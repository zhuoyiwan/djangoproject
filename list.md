# 前端功能完整优先级清单

## 1. 清单范围说明

这是一份按当前仓库代码静态对照整理出的完整版本清单，覆盖以下范围：

- 当前前端已经可访问的页面能力
- 后端已提供、但前端未接入的业务功能
- 前端已经封装 API、但页面没有使用的半成品能力
- 路由存在缺口、导航缺口、权限分层缺口
- 列表页闭环能力缺口，例如分页、排序、角色感知
- 普通 Web 前端是否需要承接的机器侧能力

这份清单的“完整”含义是：

- 对当前仓库中可识别的功能级缺口，已尽量全部覆盖
- 不包含纯视觉微调、文案润色、像素级样式问题
- 不包含未来未在仓库出现的潜在需求

## 3. P0：最高优先级，直接影响日常可用性

### P0-1. 角色感知导航与权限分层 [x]

当前问题：

- 前端导航几乎不区分角色
- 只要登录，所有用户都能看到服务器、自动化、记录等入口
- 但后端真实权限并不一致

实际后端权限：

- 审计日志：仅 `auditor` / `platform_admin`
- 自动化审批：仅 `approver` / `platform_admin`
- 自动化执行：仅 `ops_admin` / `platform_admin`
- CMDB 写操作：仅 `ops_admin` / `platform_admin`
- 用户管理：仅 `platform_admin`

缺失能力：

- 根据当前用户角色显示 / 隐藏导航项
- 根据角色显示 / 隐藏按钮
- 在进入无权限页面前就做前端拦截或降级提示
- 将“不可见”与“可见但不可操作”区分清楚

优先级原因：

- 这是平台信息架构层面的核心问题
- 不处理会导致普通用户频繁看到 403 或无效按钮
- 会直接影响“企业级产品”的可信度

---

### P0-2. 服务器资产维护闭环 [x]

当前已有：

- 列表
- 基础筛选
- 新增
- 选中详情卡

缺失能力：

- 编辑服务器
- 删除服务器
- 独立查看单台服务器完整详情
- 编辑后局部刷新或回显
- 删除后的列表联动与选中态重置

优先级原因：

- 服务器是最核心的资产对象
- 只有新增，没有维护，资产台账不闭环

---

### P0-3. 自动化高级任务查询 [x]

当前已有：

- 自动化基础列表
- 基础筛选

后端 / 前端现状：

- 后端有 `tool-query`
- 前端 API 已封装 `getJobToolQuery`
- 页面未使用

缺失能力：

- 按任务名、状态、风险、审批状态、执行器绑定等复杂查询
- 查询结果摘要
- 查询结果快速跳转任务详情

优先级原因：

- 自动化任务一多，基础表格筛选明显不够
- 该能力已经有后端与前端 API 基础，接入收益高

---

### P0-4. 自动化执行交接视图 [x]

当前已有：

- 详情页动作流转

后端 / 前端现状：

- 后端有 `handoff`
- 前端 API 已封装 `getJobHandoff`
- 页面未使用

缺失能力：

- 就绪 / 已认领任务的交接池视图
- 面向执行侧的快速处理入口
- 更清晰的 runner / 执行器关联信息

优先级原因：

- 自动化流程不只有“审批”和“详情页操作”
- 缺少交接层，执行协作体验不完整

---

### P0-5. 审计高级查询 [x]

当前已有：

- 记录列表
- 详情
- 简单关键词搜索

后端 / 前端现状：

- 后端有 `audit/logs/tool-query`
- 前端 API 已封装 `getAuditToolQuery`
- 页面未使用

缺失能力：

- 按动作查询
- 按对象查询
- 按执行人查询
- 按失败原因 / 路径 / 状态码查询
- 安全事件与业务事件快速筛选

优先级原因：

- 审计页的企业价值就在于可追溯与可过滤
- 当前只有简单搜索，远不够日常排障和审计复盘使用

---

### P0-6. 列表分页闭环 [x]

当前问题：

- 前端所有列表查询都传了 `page` 和 `page_size`
- 后端返回 `count` / `next` / `previous`
- 页面没有分页控件

影响页面：

- 服务器
- 自动化
- 记录

缺失能力：

- 下一页 / 上一页
- 页码感知
- 页大小切换
- 翻页后保持选中项与查询条件

优先级原因：

- 这是表格型后台的基础能力
- 没有分页，数据量一大就不可用

### P1-3. 自动化任务编辑 [x]

### P1-4. 列表排序能力 [x]

优先级原因：

---

### P1-5. 自动化基于角色的动作显隐 [x]

当前问题：

- 自动化详情页的按钮可用性主要按任务状态判断

- 审批人视角只突出审批动作
- 运维执行视角只突出执行动作
- 只读账号不显示高危操作按钮
- 平台管理员视角显示完整恢复动作

优先级原因：

- 这是自动化页可用性的重要组成
- 与 P0-1 相关，但值得单独落地

## 5. P2：平台能力补全

### P2-1. 用户管理页面 [x]

后端现状：

- `GET /users/`
- `GET /users/{id}/`

前端缺失：

- 用户列表
- 用户详情
- 角色信息展示
- 管理视角的账号检索入口

说明：

- 这更偏平台管理员能力
- 重要，但不是普通使用者最高频需求

---

### P2-2. 认证补全：刷新令牌 [x]

当前问题：

- 登录后拿到了 `access` 和 `refresh`
- 前端只存了 access 的使用路径
- 没有自动刷新令牌逻辑

缺失能力：

- access token 过期后的自动刷新
- 失效时的统一回退登录
- 静默续期体验

优先级原因：

- 这是长期稳定使用的基础设施能力
- 目前更像 demo 级认证体验

---

### P2-3. 注册能力是否开放 [ ]

后端现状：

- 有 `register`

前端缺失：

- 注册页面
- 邀请式注册入口
- 管理员代建账号的前端流程

建议：

- 先明确产品策略
- 如果是内部平台，未必需要公开注册页

---

### P2-4. 契约页真正落地 [x]

当前现状：

- `ContractPage.tsx` 文件存在
- `contract.ts` 数据存在
- 但无路由、无导航、用户不可访问

缺失能力：

- 挂路由
- 挂导航
- 规划该页是否仅开发态可见

说明：

- 这是典型的半成品能力
- 不一定对普通用户开放，但当前确实未完成

---

### P2-5. 审计页权限降级体验 [x]

当前问题：

- 前端允许所有已登录用户进入记录页入口
- 审计接口实际仅允许 `auditor` / `platform_admin`

缺失能力：

- 无权限用户的友好引导
- 在导航层提前隐藏或降级
- 审计页专属空态和权限说明

说明：

- 这类问题也存在于其他受限能力中
- 但审计页最明显、最值得单列

## 9. 最值得先做的 6 项

如果你要快速把这个前端从“可演示”拉到“能长期使用”，最优先的是：

1. 角色感知导航与权限分层
2. 服务器编辑 / 删除 / 完整详情
3. 自动化高级任务查询
4. 自动化执行交接视图
5. 审计高级查询
6. 服务器 / 自动化 / 记录三大列表的分页闭环

---

## 10. 结论

从当前仓库来看，前端已经具备基础主干，但仍然属于“核心链路已通、平台能力未补齐”的阶段。

最主要缺口不是单一页面，而是三类：

- 权限分层没有真正做完
- 高级查询与列表闭环没有做完
- 资产与自动化两个主模块还缺少维护型能力

把这些补齐之后，前端才算真正进入成熟的企业工作台状态。
